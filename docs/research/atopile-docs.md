## 1. What is Atopile?

**Atopile** is a tool for designing electronic circuit boards using code — bringing software development workflows to hardware design. By describing electronics with code, you can leverage modularity, version control, and deep validation.

### Philosophy

- **Code-first hardware design**: Capture design intelligence and constraints directly in code
- **Auto-selection of components**: The compiler can automatically pick components (e.g., resistors, capacitors) that meet your specifications from the JLCPCB parts library
- **Embedded calculations**: Checked on every build
- **Reliable, configurable modules**: Reusable design blocks
- **Rapid iteration**: Easier collaboration and robust designs validated through continuous integration
- **Version control**: Git-based workflow for hardware, just like software

### Tagline

> "Design circuit boards blazing fast — with code"

## 2. Installation

### Primary Method: VS Code Extension

The primary way to interact with atopile locally is via the **VS Code extension**. Also supports forks like **Cursor** or **Google Antigravity**.

1. Install a VS Code fork (VSCode, Cursor, or Google Antigravity)
2. Search for and install the **atopile extension** from the VS Code marketplace
3. Install **KiCad** for PCB layout/routing:
   - macOS: `brew install kicad`
   - Arch Linux / Ubuntu / other: see docs

### Alternative Install Methods

#### Homebrew (macOS)
```bash
brew install atopile/tap/atopile
```

#### uv (other platforms)
```bash
# Install uv first: https://docs.astral.sh/uv/getting-started/installation/
uv tool install atopile
# Verify
ato --version
```

#### Editable Installation (development)
```bash
git clone https://github.com/atopile/atopile
cd atopile
uv sync --dev
```

### Online Playground

Try atopile without installing: https://playground.atopile.io/ (nothing saved between sessions)

## 3. Project Structure

### High-Level Layout

```
<project_name>/
├─ .ato/                    # Cached dependencies (auto-managed)
├─ build/                   # Build outputs (auto-generated)
├─ ato.yaml                 # Project manifest (paths, builds, dependencies)
├─ elec/                    # Electrical files and layouts
│  ├── layouts/             # Board layout files (KiCad)
│  ├── src/                 # Core ato code and parts
│  │   ├── <project_name>.ato  # Main project file
│  │   └── parts/           # Part info (pinout, footprint, 3D model)
```

### Key Directories/Files

| Path | Purpose |
|------|---------|
| `ato.yaml` | Central manifest read by the compiler and package manager |
| `elec/src/` | Pure `.ato` source files; keep one module per file where possible |
| `elec/layouts/` | KiCad footprints, STEP models, or manual board tweaks |
| `.ato/` | Cached dependencies (auto-managed, similar to `node_modules`) |
| `build/` | Build outputs (auto-generated) |
| `elec/src/parts/` | Part definitions (pinout, footprint, 3D model files) |

### The `ato.yaml` Manifest

The manifest glues everything together. A minimal example:

```yaml
requires-atopile: "^0.10.8"

paths:
  src: ./src
  layout: ./layout

builds:
  default:                    # The target you usually build
    entry: main.ato:App
    hide_designators: true
    exclude_checks: ["PCB.requires_drc_check"]

dependencies:                 # Added automatically by `ato add`
  - type: registry
    identifier: atopile/ti-ads1115
    release: 0.1.6
```

#### Manifest Sections

- **`requires-atopile`**: Version constraint for the atopile compiler
- **`paths`**: Lets you move `src/` or `layout/` somewhere else if needed
- **`builds`**: Define one or more build targets (e.g., `default`, `panelized`, `test-jig`)
  - **`entry`**: The entry point module (format: `filename.ato:ModuleName`)
- **`dependencies`**: Registry, git, or local packages installed with `ato add`

## 5. Language Syntax (`.ato` Language)

The `ato` language is a **domain-specific language (DSL)** for describing electronic circuit boards. It's heavily inspired by and attempts to follow the syntax of **Python** for consistency and familiarity.

### Basic Types

| Type | Description |
|------|-------------|
| `module` | The primary building block — represents a circuit module |
| `interface` | Represents a connection point / signal interface |
| `Electrical` | Special built-in type of `interface` for a single electrical interface |
| `component` | Subclass of `module` that represents a single physical component |

### Defining Modules and Components

```ato
module SomeModule:
    some_signal = new ElectricSignal
    gnd = new Electrical
    some_signal.reference.lv ~ gnd
    some_variable = "some value"

# Subclassing (inheritance)
module SubclassedModule from SomeModule:
    # inherits all signals and variables from SomeModule
    some_variable = "some other value"

# Components are subclasses of modules
component Texas_Instruments_NE5532DR from LDO:
    # component-specific definitions
```

> **Note**: You can subclass a `module` as a `component`, but not the other way around.

### Creating Instances

```ato
module Test:
    gnd = new Electrical
    subclassed_module = new SubclassedModule
    subclassed_module.gnd ~ gnd
```

### Configuring Blocks

Configuration is done by assigning to attributes:

```ato
some_instance.value = 100ohm +/- 10%
```

Unlike Python, you don't need `self.` — assignments within a block scope are automatically assigned to the block.

Setting the `package` attribute makes the compiler select only components with that package:

```ato
resistor.package = "0402"
```

### Connections (the `~` operator)

Any `interface` can connect to any other interface of the same type using the `~` operator:

```ato
some_signal ~ another
power.hv ~ r_top.p1; r_top.p2 ~ output.line
```

Multiple connections can be chained on one line with `;`.

### Units

Units are written (without a space) after any number:

```ato
1.23Ω        # or 1.23ohm
4.56Kohm     # kilohms
7.89Mohm     # megohms
23.4uF       # microfarads
```

Supported unit types: resistance (ohm/Ω), capacitance (F), voltage (V), current (A), degrees, etc.

### Tolerances

```ato
1V to 2V           # range
3uF +/- 1uF        # absolute tolerance
4Kohm +/- 1%       # percentage tolerance
```

### Combined Units and Tolerances

```ato
3V to 3.6V          # supply rail
3V +/- 10mV         # reference
4.7uF +/- 20%       # generic cap
```

### Assertions and Math

Use `assert` to apply constraints on values:

```ato
a = 1 ± 0.1
b = 2 ± 0.2
c: resistance                    # variable declaration without assignment

assert a < b                     # comparison assertion
assert c within 1Kohm to 10Kohm  # range constraint (solved for, then checked)
```

Supported operators: `<`, `>`, `within` (all inclusive of bounds).

The `assert ... is` form links variables:

```ato
assert v_out is v_in * r_bottom.resistance / r_total
assert v_out is output.reference.voltage
```

atopile automatically **solves systems of constraints** for free variables and checks that values are within tolerances.

### Variable Declarations

```ato
v_in: voltage
v_out: voltage
max_current: current
total_resistance: resistance
ratio: dimensionless
```

### Specialization (the `->` operator)

Specialize a module from its existing instance to another type:

```ato
some_instance -> AnotherModuleType
```

### Imports

```ato
from "where.ato" import What, Why, Wow
```

- The file path is a string (in quotes)
- Import names must match precisely (case-sensitive)
- Paths are relative to the project root (where `ato.yaml` is) or within the standard library (`.ato/modules/`)

**Shorthand import** (for standard library):

```ato
import Resistor
import Resistor, ElectricPower, ElectricSignal
```

> **Warning**: Legacy syntax `import XYZ from "abc.ato"` is deprecated.

### Documentation (Docstrings)

```ato
module VoltageDivider:
    """
    A voltage divider using two resistors.
    Connect to the `power` and `output` interfaces
    Configure via:
    - `power.voltage`
    - `output.reference.voltage`
    - `max_current`
    """
```

### Comments

```ato
# This is a comment
r1 = new Resistor  # Inline comment
```

### Signals and Pins

```ato
signal IN1neg ~ pin 2    # Create signal and map to footprint pad
signal OUT1 ~ pin 1
```

The `pin` keyword creates a signal and connects it to a pad on the footprint.

### Traits

Traits are metadata annotations on components:

```ato
#pragma experiment("TRAITS")
import has_datasheet_defined
import has_designator_prefix
import has_part_picked
import is_atomic_part
import is_auto_generated

component Texas_Instruments_NE5532DR_package:
    """Dual Low-Noise High-Speed Audio Operational Amplifier"""
    trait is_auto_generated<system="ato_part", source="easyeda:C7426", ...>
    trait is_atomic_part<manufacturer="Texas Instruments", partnumber="NE5532DR", footprint="SOIC-8_L4.9-W3.9-P1.27-LS6.0-BL.kicad_mod", symbol="NE5532DR.kicad_sym", model="SOIC-8_L4.9-W3.9-H1.7-LS6.0-P1.27.step">
    trait has_part_picked::by_supplier<supplier_id="lcsc", supplier_partno="C7426", manufacturer="Texas Instruments", partno="NE5532DR">
    trait has_designator_prefix<prefix="U">
    trait has_datasheet_defined<datasheet="https://...">
```

## 6. Build Process

### Running a Build

```bash
ato build
# or with verbose output:
ato -v build
```

When you run `ato build`:

1. The compiler reads your `.ato` source files
2. Resolves imports and dependencies
3. Solves constraint systems (assertions, tolerances)
4. Auto-selects components from the JLCPCB parts library
5. Generates a KiCad project file (`.kicad_pcb`)
6. Opens KiCad if building a single target

### Build Outputs

After a successful build, you can visualize your design:

- **2D Layout**: Built-in viewer showing component placement and board outline
- **3D Model**: Built-in 3D preview of assembled board
- **KiCad**: Opens layout in KiCad for routing and detailed editing

### Manufacturing Export

The manufacturing export wizard generates:

| Output | Format | Purpose |
|--------|--------|---------|
| Gerber files | `.zip` | PCB layer artwork for fabrication |
| BOM | `.csv` | Component list for procurement |
| Pick and place | `.csv` | Component placement data for assembly |

The exported files are ready to upload to **JLCPCB**.

```
ato [app-options] command [command-options] arguments
```

## 8. Package Management

### Finding Packages

- **First-party packages**: https://packages.atopile.io/
- **Community packages**: Available on GitHub and other repositories

### Using Packages

```bash
# Add a package
ato add atopile/ti-ads1115

# Then import in your .ato file
from "ti-ads1115.ato" import TI_ADS1115

module ExampleADCProject:
    adc = new TI_ADS1115
```

### Managing Dependencies

Dependencies are stored in `ato.yaml`:

```yaml
dependencies:
  - type: registry
    identifier: atopile/buttons
    release: 0.2.2
  - type: registry
    identifier: atopile/ti-ads1115
    release: 0.1.6
```

### Dependency Commands

| Command | Action |
|---------|--------|
| `ato sync` | Install all dependencies from `ato.yaml` (run when pulling a new project) |
| `ato add <package>` | Add package to `ato.yaml`, install locally, sync all |
| `ato remove <package>` | Remove from `ato.yaml`, remove locally, sync remaining |

### Development Dependencies

```bash
# From git
ato add git://{git-url}

# From local directory
ato add file://./path/to/package
```

> **Note**: You cannot publish a package that depends on unpublished packages.

Dependencies are cached in the project's `.ato/` directory (similar to `node_modules` or Python's `.venv`).

## 9. Integration with Other Tools

### KiCad

- atopile uses **KiCad** (open source) for PCB layout
- `ato build` generates a KiCad project file
- The `ato` compiler automatically installs a **KiCad plugin** for layout sync
- The plugin supports "Sync Group" to pull layouts from imported packages
- Run `ato configure` to re-install the plugin if needed

### JLCPCB

- Component auto-selection uses the **JLCPCB parts library**
- Manufacturing export generates files ready for **JLCPCB upload**
- Export includes Gerber, BOM, and pick-and-place files

### faebryk

- atopile integrates with **faebryk** (a Python-based EDA framework)
- The `Electrical` type is a built-in from faebryk

### Git / Version Control

- atopile is designed for git-based workflows
- Standard git workflow: branch → commit → push → pull request
- `.ato/` directory should be in `.gitignore` (cached dependencies)
- `build/` directory should be in `.gitignore` (generated outputs)

## 10. Example Projects

### Minimal Hello World

```ato
import Resistor

module App:
    r1 = new Resistor                    # Create a new resistor
    r1.resistance = 50 kohm +/- 10%      # Set the resistor value
```

### Voltage Divider (Simple)

```ato
import Resistor

module VoltageDivider:
    r_top = new Resistor
    r_bottom = new Resistor
    r_top.p2 ~ r_bottom.p1

module App:
    my_vdiv = new VoltageDivider
    my_vdiv.r_top.resistance = 10kohm +/- 10%
    my_vdiv.r_bottom.resistance = 4.7kohm +/- 10%
```

### Voltage Divider (Production Quality)

```ato
import Resistor, ElectricPower, ElectricSignal

module VoltageDivider:
    """
    A voltage divider using two resistors.
    Connect to the `power` and `output` interfaces
    Configure via:
    - `power.voltage`
    - `output.reference.voltage`
    - `max_current`
    """
    # External interfaces
    power = new ElectricPower
    output = new ElectricSignal

    # Components
    r_bottom = new Resistor
    r_top = new Resistor

    # Variables
    v_in: voltage
    v_out: voltage
    max_current: current
    total_resistance: resistance
    ratio: dimensionless
    r_total: resistance

    # Connections
    power.hv ~ r_top.p1; r_top.p2 ~ output.line
    output.line ~ r_bottom.p1; r_bottom.p2 ~ power.lv

    # Link interface voltages
    assert v_out is output.reference.voltage
    assert v_in is power.voltage

    # Equations
    assert r_top.resistance is (v_in / max_current) - r_bottom.resistance
    assert r_bottom.resistance is (v_in / max_current) - r_top.resistance
    assert r_top.resistance is (v_in - v_out) / max_current
    assert r_bottom.resistance is v_out / max_current
    assert r_bottom.resistance is r_total * ratio
    assert r_top.resistance is r_total * (1 - ratio)

    # Calculate outputs
    assert r_total is r_top.resistance + r_bottom.resistance
    assert v_out is v_in * r_bottom.resistance / r_total
    assert v_out is v_in * ratio
    assert max_current is v_in / r_total
    assert ratio is r_bottom.resistance / r_total

module App:
    my_vdiv = new VoltageDivider
    assert my_vdiv.power.voltage is 10V +/- 1%
    assert my_vdiv.output.reference.voltage within 3.3V +/- 10%
    assert my_vdiv.max_current within 10uA to 100uA
```

## 11. Adding Components (Four Methods)

### Method 1: Auto-Pick Passive Components

Works for resistors, capacitors, and inductors. The compiler selects from JLCPCB parts library:

```ato
resistor = new Resistor
resistor.resistance = 10kohm +/- 5%
resistor.package = "0402"
```

### Method 2: Add Specific Component via CLI

```bash
ato create part
# Enter JLCPCB part number (e.g., C7426) or exact MPN (e.g., NE5532DR)
```

This generates a `.ato` file with pinout, footprint, and 3D model references.

### Method 3: Use a Package from the Registry

```bash
ato add atopile/rp2040
```

### Method 4: Create Custom Part Manually

1. Create a folder in `project/parts/` named after your component
2. Add KiCad footprint and 3D model files
3. Create a `.ato` file with the component definition

## 13. Typical Workflow

1. Sketch your circuit on paper
2. Search https://packages.atopile.io and GitHub for pre-existing modules
3. Use `ato add` to install packages
4. Design modules and calculations using `ato` code
5. Run `ato build` to compile, choose components, update layout
6. Use KiCad to lay out changes
7. Repeat steps 4-6 until satisfied
8. Push changes to git repo
9. CI automatically builds, tests, and generates manufacturing files
10. Download gerbers from CI
11. Order PCBs from manufacturer (e.g., JLCPCB)
